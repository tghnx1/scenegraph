API Reference

Current backend API for the local Docker stack. The public entry point is nginx at http://localhost:8080; nginx proxies /api/* and /health to the FastAPI backend.

Real Postgres data Read-only runtime endpoints No auth yet No pagination yet, only graph limit
GET/health

Checks that the API is running and can query Postgres.

GET/api

Returns a simple backend status message.

GET/api/venues

Returns imported venue rows mapped into the frontend's current venue shape.

GET/api/graph

Returns graph nodes and links for imported events, artists, venues, and genres.

Base URL

Use the nginx URL during local development:

http://localhost:8080

Inside Docker, the backend listens on backend:3000, but that is not the public browser entry point.

GET /health

Returns API and database health. Docker also uses this endpoint for backend healthchecks.

curl http://localhost:8080/health
{
  "status": "ok",
  "database": "ok"
}

GET /api

Small root endpoint for confirming that nginx reaches FastAPI.

curl http://localhost:8080/api
{
  "message": "Berlin Scene Graph backend is running."
}

GET /api/venues

Returns venues from the imported Postgres schema. The backend maps imported RA venue columns into the current frontend naming:

Response field Source
id venues.id
name venues.name
district COALESCE(area_name, country_code, '')
scene_focus COALESCE(address, content_url, '') 
curl http://localhost:8080/api/venues
{
  "venues": [
    {
      "id": 1,
      "name": "Example Venue",
      "district": "Berlin",
      "scene_focus": "https://ra.co/clubs/example"
    }
  ]
}

GET /api/graph

Returns a graph payload for the main frontend graph view. The endpoint reads real Postgres data from events, venues, event_genres, genres, event_artists, and artists.

Query parameters

Name Type Default Description
genre string none Filters events by exact genre name, case-insensitive. Empty strings are rejected.
dateFrom date, YYYY-MM-DD none Includes events on or after this date.
dateTo date, YYYY-MM-DD none Includes events on or before this date.
limit integer, 1 to 1000 500 Limits the number of selected event rows before graph expansion.

Example request

curl "http://localhost:8080/api/graph?genre=techno&dateFrom=2026-04-01&dateTo=2026-04-30&limit=100"

Validation and empty results

  • If dateFrom is later than dateTo, the API returns 400.
  • If limit is outside 1..1000, FastAPI returns 422.
  • If filters match no events, the API returns {"nodes": [], "links": []}.
{
  "detail": "dateFrom must be earlier than or equal to dateTo."
}

Current Gaps

  • No authentication or roles are implemented, so there are no protected API routes yet.
  • No public write endpoints exist for imports. Imports are local operator tasks through make full-pipeline.
  • Artist profile, similar artists, and artist ego graph are still frontend mock-backed.
  • Promoters are imported by the data script, but /api/graph does not emit promoter nodes or links yet.
  • Advanced search, sorting, offset/cursor pagination, chat, friends, and WebSocket features are not implemented yet.

Operator Commands

These are not public API endpoints, but they are part of the current local workflow.

make upd
make prisma-migrate
make full-pipeline
make import-dump DUMP=/absolute/path/to/scenegraph_dump.sql
RESET_DB=1 make import-dump DUMP=/absolute/path/to/scenegraph_dump.sql
make db-shell

make import-dump creates the configured database if it is missing. If the database already exists, it asks before deleting and recreating it. Use RESET_DB=1 only when you intentionally want to skip the prompt.

Dump Import vs Incremental Import

Use the SQL dump workflow when a teammate needs the same database snapshot as another teammate. A dump is treated as a full restore, not as an incremental patch.

Workflow Command When to use it
Full snapshot restore make import-dump DUMP=/absolute/path/to/scenegraph_dump.sql Use when sharing a complete local database state with the team. If the database exists, the script asks before overwriting it.
Incremental/upsert import make full-pipeline Use when running the full local scrape/import/enrichment flow without replacing the whole database from a dump.

Plain SQL dumps usually contain CREATE TABLE, INSERT, or COPY statements. Replaying them into an existing database can fail on existing tables, duplicate keys, or duplicate rows. For incremental imports, use the validated importer because it writes with upsert behavior. If SQL dumps ever become the required incremental exchange format, restore the dump into a staging database first, then merge into the main database with explicit INSERT ... ON CONFLICT DO UPDATE statements.
