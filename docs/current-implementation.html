This deck documents the live Step 1 implementation as it exists in the repository today: a single external gateway through nginx, a React/Vite frontend, a FastAPI backend, and a Postgres database with a tiny bootstrap data path.

Traffic Diagram

Request Flow Through the Stack

The browser never talks directly to the FastAPI container or the database. It only talks to nginx, and nginx routes requests internally.

External Client Browser

Requests http://localhost and gets one same-origin app surface.

→
Published Port nginx

Listens on port 80 and decides whether traffic goes to frontend or backend.

UI Route frontend

Receives / and serves the React/Vite app.

↘
API Route backend

Receives /health and /api/* requests through nginx.

→
Internal Data db

Only the backend talks to Postgres over the internal Docker network.

How nginx routes now

  • / goes to the Vite frontend service.
  • /health is proxied to FastAPI.
  • /api/... is proxied to FastAPI.
  • The browser sees one origin, so current reads avoid direct cross-origin calls.

Why this matters

  • Only one service is published to the host.
  • Backend and database are hidden from the host network.
  • Later HTTPS and auth flow can land in nginx instead of every container.

Startup Order

What Happens During Docker Startup

The stack uses Docker healthchecks and dependency ordering so services come up in a predictable sequence instead of racing each other.

1

Compose reads .env

Variables such as DATABASE_URL and Postgres credentials are loaded before container creation.

2

Postgres starts first

The db service becomes healthy only after pg_isready succeeds.

3

FastAPI waits for the database

The backend service depends on healthy Postgres before it starts.

4

Backend bootstrap runs on startup

FastAPI lifespan calls the temporary DB bootstrap: create venues if missing, then seed demo rows if the table is empty.

5

Backend healthcheck starts polling

Docker runs a tiny Python command that requests http://127.0.0.1:3000/health inside the container.

6

Frontend starts, then nginx starts last

nginx depends on backend health and frontend startup, so the public gateway comes online after the inner services are ready.

Deterministic startup beats lucky startup Healthchecks keep the boot flow honest

Current Truth

What Is Deliberate Now and What Must Change Later

The current stack is intentionally split between decisions that are stable enough to keep and temporary decisions that only exist to make Step 1 demonstrable.

Keep for now

  • Single external gateway through nginx.
  • Internal-only backend and Postgres exposure.
  • FastAPI health endpoint and Docker healthchecks.
  • One visible read-only slice proving real integration.
  • Named Postgres volume for local persistence.

Replace later

  • Startup SQL that auto-creates the demo table.
  • Seed rows inserted directly by backend startup logic.
  • Single-file backend route layout in main.py.
  • Demo-only venues bootstrap once the real shared schema arrives.
  • Ad hoc bootstrap data path once migrations and import flows exist.

Current cleanup rule

As soon as the team lands the shared schema for venues, promoters, artists, and events, the backend startup SQL should be removed and replaced by the real migration flow.

Temporary does not mean accidental Documented tradeoffs reduce future confusion
