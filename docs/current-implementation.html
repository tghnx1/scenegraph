The browser never talks directly to the FastAPI container or the database. It only talks to nginx, and nginx routes requests internally.

Internal Data db

Only the backend talks to Postgres over the internal Docker network.

How nginx routes now

  • / goes to the Vite frontend service.
  • /health is proxied to FastAPI.
  • /api/... is proxied to FastAPI.
  • The browser sees one origin, so runtime API reads do not need direct backend exposure.

Why this matters

  • Only one service is published to the host.
  • Backend and database are hidden from the host network.
  • Later HTTPS and auth flow can land in nginx instead of every container.

Database and Import

How Data Gets Into the Runtime Graph

The backend no longer creates demo tables on startup. Schema changes live in Prisma migrations, and event data is loaded through a validation script.

1

Schema is defined under backend/prisma

backend/prisma/schema.prisma and its migrations define the shared Postgres tables for events, venues, artists, promoters, genres, and relations.

2

Migrations run through the Prisma tool container

make prisma-migrate runs the Compose prisma profile and applies migrations against the Docker Postgres database.

3

Source data stays local

Import files are expected under backend/data. The directory is mounted into the backend container and ignored by git except for .gitkeep.

4

Importer validates before writing

backend/scripts/import_events.py checks required event, venue, artist, promoter, genre, and image fields before upserting rows.

5

Runtime API reads from Postgres

/api/graph joins imported events, venues, genres, and artists to return graph nodes and links.

6

Postgres persists in a named volume

The database files live inside Docker's named volume scenegraph_postgres_data, mounted as /var/lib/postgresql/data in the db container.

Schema through Prisma, data through importer, reads through FastAPI Database-backed graph
Backend Routes

Current Public API Surface

The public API is intentionally small. It exposes health, a root message, venues, and the graph data needed by the current UI.

Routes

  • GET /health checks database connectivity.
  • GET /api returns a backend status message.
  • GET /api/venues returns imported venue rows.
  • GET /api/graph returns artist, event, and venue nodes with graph links.

Graph filters

  • genre filters events by exact genre name, case-insensitive.
  • dateFrom includes events on or after a date.
  • dateTo includes events on or before a date.
  • limit caps selected events, default 500, max 1000.

Example

GET /api/graph?genre=techno&dateFrom=2026-04-01&dateTo=2026-04-30&limit=100

The response contains nodes and links. Current links are artist → event as performed_at and event → venue as held_at.

Current Truth

What Is Real Now and What Is Still Mocked

The project has moved past the first demo database slice, but not every frontend surface has a backend route yet.

Real now

  • Single external gateway through nginx.
  • Internal-only backend and Postgres exposure.
  • FastAPI health, venues, and graph endpoints.
  • Main frontend graph reads from /api/graph.
  • Validated import into the Prisma-defined Postgres schema.
  • Named Postgres volume for local persistence.

Still mocked or not implemented

  • Artist profile and similar artist calls still use frontend/src/api/mock_data.
  • Artist ego graph is mock-backed until /api/graph/artist/{id} exists.
  • Promoter tables are imported, but promoter graph nodes and links are not emitted yet.
  • No authentication, friends, chat, or WebSocket layer exists yet.
  • No recommendation engine or advanced search endpoint exists yet.

Current cleanup rule

Keep mock-backed frontend calls clearly separated from real API calls. When a backend route is added, replace the mock adapter in frontend/src/api and update the API documentation in the same change.
