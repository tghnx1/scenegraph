Slide 2 / What we are building

What kind of product is this, and why is it interesting?

In the project docs, the product is described as a web app for emerging DJs. It builds a map of the local scene from event data, links artists, promoters, venues, and events, and then explains where a user can move next and why.

Problem
The music scene is opaque. A newcomer has fragments of information: who played where, who organizes what, which styles overlap. But there is no real system-level map.
Solution
We transform scattered events into a graph: Artist, Event, Promoter, Venue. Then the system ranks “next steps” not by fame, but by reachability.
Key thesis
Not “who is most popular,” but “who is realistically close to your current position in the scene,” and through which connections we can prove that.
User journey
Step 1
Profile
Account, scene interests, links, and artist identity.
Step 2
Mapping
The system finds the nearest graph nodes and the user’s starting position.
Step 3
Ranking
The app scores nearby promoters, venues, artists, and events.
Step 4
Explanation
The system shows the path: through whom, through what, and across how many hops.
What “reachability” means
  • A very large and famous promoter may be stylistically relevant, but practically unreachable.
  • A smaller or mid-level promoter with overlap in lineup, venue, and style is often much more useful.
  • That is why the recommendation engine must account for graph proximity, level gap, and repeated connections.
  • This makes the product explainable and useful, not just visually impressive in a screenshot.
Product lens: reachability over popularity

The most important lesson from the docs is that order matters. We cannot start from the “smart graph” until there is a defendable web platform that passes the mandatory part and safely collects module points.

  • Realtime features — major.
  • User interaction — major.
  • Public API — major.
  • Analytics dashboard — major.
  • Custom scene graph recommendation engine — major.
  • ORM, advanced search, browser support, export/import — minor.
What we do not build too early

Slide 5 / Communication map

How the pieces talk to each other and through what

This is one of the most important slides for the team because it removes the magic. These are not abstract blocks, but real communication types: HTTPS, WebSocket, ORM/SQL, batch ingestion, and explanation payloads.

Runtime flow in plain language
  • Browser → API: login, profile, search, dashboard, recommendation requests go over HTTPS.
  • Browser ↔ WebSocket hub: chat, presence, notifications and optional refresh events go over a persistent socket connection.
  • API → DB: backend reads user state, graph entities, metrics and recommendations through an ORM or direct query layer.
  • API → Recommendation engine: engine calculates ranked candidates and builds human-readable explanation paths.
  • Recommendation engine → DB: reads precomputed metrics and graph relationships, not raw scraped pages.
  • Offline pipeline → DB/import layer: event data and enrichments are prepared ahead of time, so runtime stays fast and defendable.

Slide 6 / Offline ingestion

Data pipeline: how the scene enters the product

This is the part where the existing docs and code are especially helpful. The pipeline is already designed in an event-first way, and the current repo already implements a meaningful part of it as a parser chain.

How to explain the pipeline to the team
  • Step 1: we collect events because the event is the best container of context: lineup, date, promoter, venue, and description.
  • Step 2: we derive entities from events, not the other way around. That protects us from “dead” artists with no scene around them.
  • Step 3: we pull biographies and style context separately so the recommendation engine can explain matches through more than co-occurrence alone.
  • Step 4: we precompute metrics in advance. Runtime should not parse the internet and should not wait for heavy calculations.
  • Step 5: the result of ingestion is not just JSON files, but prepared graph state for application runtime.

Slide 7 / Graph model

What entities and relations actually form the scene

The base MVP graph is intentionally compact. The core documents converge on four main entities: Artist, Event, Promoter, and Venue. In an extended version we can add Label and Radio, but not too early.

Why these four nodes are enough for an MVP
  • Artist gives us career trajectory, style, and lineup neighbors.
  • Event is the key contextual node where the other entities meet.
  • Promoter is the actor that actually creates opportunity, not just an abstract brand.
  • Venue helps us understand scene level, recurring nights, and geographic or cultural niche.
MVP graph: Artist -> Event Promoter -> Event Event -> Venue Extended graph later: Artist -> Label Artist -> Radio Promoter -> Collective links Venue -> recurring series
Compact graph first, richer graph later
Every extra entity must earn its maintenance cost
Slide 8 / Recommendation logic

How the recommendation engine makes decisions

According to the docs, the recommendation logic is built from five major signals: proximity, strength, style similarity, relevance, and reachability. This is not a black box and not “AI magic,” but an explainable scoring pipeline.

High-level scoring model
score = w1 * graph_proximity + w2 * connection_strength + w3 * style_similarity + w4 * scene_relevance + w5 * reachability
The exact equation matters less than the fact that every component is explainable and can be shown to the user in the UI.

Slide 9 / 42 module strategy

Which modules we should actually claim, and why this is defendable

According to the docs, the project has a safe target: 16 points without depending on AI-related modules. That matters because even if the recommendation core becomes strong, pass/fail should not depend on a debatable reading of the subject.

Major modules
  • Framework for frontend + backend
  • Realtime features
  • User interaction
  • Public API
  • Advanced analytics dashboard
  • Scene graph recommendation & explanation engine
Minor modules
  • ORM
  • Advanced search
  • Additional browser support
  • Data export / import
Why this is safe
  • Each module has clear user-facing or evaluator-facing proof.
  • The modules can be demonstrated independently: chat, API, search, dashboard, export.
  • The custom module is central project logic, not a cosmetic feature.
What we do not claim too early
  • AI points as a critical dependency.
  • ML, embeddings, or RAG if they are not fully defendable.
  • Large scope additions like organizations, SSR, or microservices.
Safe 16-point plan beats fuzzy ambition
Everything claimed must be easy to prove live

Slide 11 / Step 1 sprint plan

Step 1 only: ship the walking skeleton

Step 1 is not the recommendation engine. Step 1 is one read-only vertical slice: the app boots, the backend talks to the database, real seed data is imported, and one page shows real artists or events from the Berlin scene dataset.

Day 1
Repo and app skeleton
Finalize repo structure, boot frontend shell, boot backend shell, define env files, and agree on one command sequence for local startup.
Day 2
First schema and migrations
Create only the minimum tables for Step 1: users, artists, events, promoters, venues. Keep it simple and import-friendly.
Day 3
Seed import bridge
Take a tiny clean subset from the current parser output and write a reproducible import script that fills the DB with demo-ready records.
Day 4
Read-only API
Expose a health route plus one real data route such as GET /artists or GET /events with simple pagination or a limit.
Day 5
First visible page
Render one clean read-only page such as Browse Artists or Recent Events with loading, empty, and error states.
Day 6
Integration and cleanup
Wire frontend to backend, backend to DB, and seed import to real parser data. Remove mocks and fix basic DX gaps.
Day 7 demo checklist
  • The app opens locally without guesswork.
  • The database migration runs cleanly from scratch.
  • The seed script imports 20–50 real records reproducibly.
  • One backend route returns real DB data, not mocks.
  • One frontend page displays that real scene data end-to-end.
Definition of done
Step 1 is done when the team can honestly say:

“We already have a real web app connected to a real database, populated from our parser pipeline, and serving real scene data through the backend.”

No auth yet. No graph edges yet. No recommendations yet. Just one stable vertical slice.
Step 1 is about proof of integration, not product ambition
If this slice is unstable, everything later will wobble

Slide 13 / Risks and anti-patterns

Where the project can very realistically go wrong

This slide is not about fear. It is about calm. The earlier the team names the real risks, the less likely the project is to break from scope creep or from trying to be impressive instead of defendable.

Risk 1
Starting from the graph instead of the platform.
Then, during defense week, you discover that the logic is interesting but login, realtime, API, or browser support are shaky.
Risk 2
Claiming modules that are too “smart.”
If AI, RAG, or ML are not fully provable, they usually hurt more than they help. Safe modules matter more.
Risk 3
Overestimating the data pipeline.
Scraping, enrichment, and style tagging grow very easily. Runtime must still survive with a limited seed dataset.
Risk 4
Making a beautiful UI without explainability.
If a recommendation cannot be explained through a path and facts, an evaluator may treat the feature as magical and unproven.
Mitigation
Ship vertical slices, not massive branches.
Mitigation
Keep a stable demo dataset independent from live scraping.
Mitigation
Document every claimed module with visible proof in UI or API.
Biggest trap: fancy complexity before stable proof
Defense likes clarity more than maximal cleverness
Slide 14 / Final demo

What the live project demo should look like

A strong Transcendence demo is not random clicking. It is a deliberately constructed narrative where every screen proves part of the architecture and part of the module strategy.

Recommended live flow
  • Show signup/login and quickly prove auth plus protected routes.
  • Open the user profile and scene preferences.
  • Open search and show filtering for artists, events, promoters, and venues.
  • Open the graph/recommendation page and show top promoters, venues, artists, and events.
  • Click a recommendation and show the explanation path step by step.
  • Show chat, presence, and notifications as proof of realtime features.
  • Show the dashboard and export as proof of analytics modules.
  • Briefly show the public API or docs page.
What evaluator should remember
  • This is a complete web product, not only a data experiment.
  • The custom module is truly central: the recommendation engine is explainable and project-specific.
  • Realtime, search, API, dashboard, and the social layer are not decorative. They are built into the product scenario.
  • The team understands the architecture, not just “something that happened to work.”
