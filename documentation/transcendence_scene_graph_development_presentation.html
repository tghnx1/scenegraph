Slide 2 / What we are building

Что это за продукт и почему он вообще интересен

В документации продукт описан как web-приложение для emerging DJs. Оно строит карту местной сцены из event data, связывает артистов, промоутеров, площадки и события, а потом объясняет, куда двигаться дальше и почему именно туда.

Проблема
Музыкальная сцена непрозрачна. У новичка есть куски информации: кто где играл, кто что организует, какие стили совпадают. Но нет системной карты.
Решение
Мы превращаем разрозненные события в graph: Artist, Event, Promoter, Venue. Затем система ранжирует "следующие шаги" не по fame, а по reachability.
Ключевой тезис
Не "кто самый популярный", а "кто реально близок к твоему текущему положению в сцене" и через какие связи это видно.
Пользовательский цикл
Step 1
Профиль
Аккаунт, scene interests, ссылки, artist identity.
Step 2
Mapping
Система находит ближайшие узлы графа и стартовую позицию.
Step 3
Ranking
Считаются promoters, venues, artists, events рядом по сцене.
Step 4
Explanation
Показывается путь: через кого, через что, через сколько hops.
Что значит "reachability"
  • Слишком большой и знаменитый promoter может быть stylistically relevant, но practically unreachable.
  • Малый или средний promoter с пересечением по lineup, venue и style часто гораздо полезнее.
  • Поэтому recommendation engine должен учитывать и близость по graph, и level gap, и повторяемость связей.
  • Это делает продукт объяснимым и полезным, а не просто впечатляющим на скриншоте.
Product lens: reachability over popularity

Как думать о проекте, если за спиной только School 42

Самая важная мысль из документации: порядок matters. Нельзя начинать с "умного графа", пока не существует defendable web-platform, которая проходит mandatory и безопасно собирает очки по модулям.

Layer A
Mandatory skeleton.
Docker, frontend shell, backend shell, auth, DB, HTTPS path, validation, privacy/terms, multi-user sanity.
Layer B
Point-safe modules.
ORM, user interaction, realtime, public API, search, browser support, analytics dashboard, export/import.

Core differentiator.

  • Realtime features — major.
  • User interaction — major.
  • Public API — major.
  • Analytics dashboard — major.

  • ORM, advanced search, browser support, export/import — minor.

  • Не строим AI-first demo ради впечатления, если базовые модули еще не закрыты.

Slide 5 / Communication map

Как что с чем общается и через что именно

Это один из самых важных слайдов для команды: он убирает магию. Ниже не просто "блоки", а реальные типы коммуникации: HTTPS, WebSocket, ORM/SQL, batch ingestion и explanation payloads.

Runtime flow in plain language
  • Browser → API: login, profile, search, dashboard, recommendation requests go over HTTPS.
  • Browser ↔ WebSocket hub: chat, presence, notifications and optional refresh events go over a persistent socket connection.
  • API → DB: backend reads user state, graph entities, metrics and recommendations through an ORM or direct query layer.
  • API → Recommendation engine: engine calculates ranked candidates and builds human-readable explanation paths.
  • Recommendation engine → DB: reads precomputed metrics and graph relationships, not raw scraped pages.
  • Offline pipeline → DB/import layer: event data and enrichments are prepared ahead of time, so runtime stays fast and defendable.

Slide 6 / Offline ingestion

Data pipeline: как сцена попадает в продукт

Здесь особенно полезно опираться на уже существующую документацию и текущий код. У вас pipeline уже мыслится event-first, а в текущем repo он даже частично реализован как цепочка парсеров.

Как объяснить pipeline команде
  • Шаг 1: мы собираем события, потому что event — лучший контейнер контекста: lineup, date, promoter, venue, description.
  • Шаг 2: из событий выводим entities, а не наоборот. Это защищает от "мертвых" артистов без сцены вокруг них.
  • Шаг 3: отдельно тянем biographies и style context, чтобы потом recommendation engine мог объяснять совпадения не только по co-occurrence.
  • Шаг 4: precompute metrics заранее. Runtime не должен парсить интернет и не должен ждать тяжелых вычислений.
  • Шаг 5: результат ingestion — это не просто JSON-файлы, а подготовленное graph state для app runtime.

Slide 7 / Graph model

Какие сущности и связи реально формируют сцену

Базовый MVP-граф намеренно компактный. Основные документы сходятся на четырех core entities: Artist, Event, Promoter, Venue. В расширенной версии можно добавить Label и Radio, но не нужно делать это слишком рано.

Почему именно эти 4 узла достаточно хороши для MVP
  • Artist дает нам карьерную траекторию, стиль и соседей по lineup.
  • Event это ключевой контекстный узел: в нем встречаются все остальные сущности.
  • Promoter — это тот, кто реально создает opportunity, а не просто существует как абстрактный бренд.
  • Venue помогает понять уровень сцены, recurring nights и географическую/культурную нишу.
MVP graph: Artist -> Event Promoter -> Event Event -> Venue Extended graph later: Artist -> Label Artist -> Radio Promoter -> Collective links Venue -> recurring series

Slide 8 / Recommendation logic

Как recommendation engine принимает решения

По документам recommendation logic строится из пяти крупных сигналов: proximity, strength, style similarity, relevance и reachability. Это не black box и не "магия AI", а объяснимый scoring pipeline.

High-level scoring model
score = w1 * graph_proximity + w2 * connection_strength + w3 * style_similarity + w4 * scene_relevance + w5 * reachability
Важно не точное уравнение, а то, что каждая составляющая объяснима и может быть показана пользователю в UI.

Slide 9 / 42 module strategy

Какие модули мы реально заявляем и почему это defendable

По документации у проекта есть безопасная цель: 16 очков без зависимости от AI-модулей. Это важно: если recommendation core выйдет сильным — отлично, но pass/fail не должен зависеть от спорной интерпретации subject.

Major modules
  • Framework for frontend + backend
  • Realtime features
  • User interaction
  • Public API
  • Advanced analytics dashboard
  • Scene graph recommendation & explanation engine
Minor modules
  • ORM
  • Advanced search
  • Additional browser support
  • Data export / import
Почему это безопасно
  • Каждый модуль имеет четкий user-facing или evaluator-facing proof.
  • Модули можно демонстрировать независимо: chat, API, search, dashboard, export.
  • Custom module — центральная логика проекта, а не cosmetic feature.
Что не заявляем рано
  • AI points как критическую зависимость.
  • ML / embeddings / RAG, если они не полностью defendable.
  • Большие scope additions вроде organizations, SSR, microservices.
Safe 16-point plan beats fuzzy ambition
Everything claimed must be easy to prove live

Slide 13 / Risks and anti-patterns

Где проект почти наверняка может поехать не туда

Этот слайд нужен не для страха, а для спокойствия. Чем раньше команда проговаривает реальные риски, тем меньше шансов, что проект сломается от scope creep или от попытки "сделать впечатляюще" вместо "сделать defendable".

Risk 1
Начать с графа, а не с платформы.
Тогда на неделе защиты выяснится, что интересная логика есть, а login, realtime, API или browser support shaky.
Risk 2
Claim слишком умные модули.
Если AI/RAG/ML не полностью доказуемы, они скорее вредят, чем помогают. Safe modules важнее.
Risk 3
Переоценить data pipeline.
Scraping, enrichment и style tagging легко разрастаются. Runtime должен жить даже при ограниченном seed dataset.
Risk 4
Сделать красивый UI без explainability.
Если recommendation нельзя объяснить path'ом и фактами, evaluator может считать фичу "магической и недоказанной".
Mitigation
Ship vertical slices, not massive branches.
Mitigation
Keep a stable demo dataset independent from live scraping.
Mitigation
Document every claimed module with visible proof in UI or API.
Biggest trap: fancy complexity before stable proof
Defense likes clarity more than maximal cleverness
Slide 14 / Final demo

Как должна выглядеть живая демонстрация проекта

Хороший demo flow для Transcendence — это не "тыкать куда попало". Это заранее выстроенная narrative, где каждый экран доказывает кусок architecture и кусок module strategy.

Recommended live flow
  • Показать signup/login и быстро доказать auth + protected routes.
  • Открыть профиль пользователя и scene preferences.
  • Открыть search и показать фильтрацию artists/events/promoters/venues.
  • Открыть graph/recommendation page и показать top promoters, venues, artists, events.
  • Кликнуть по рекомендации и показать explanation path step by step.
  • Показать chat / presence / notification как proof realtime.
  • Показать dashboard и export как proof analytics modules.
  • Коротко показать public API или docs page.
What evaluator should remember
  • Это полноценный web product, не только data experiment.
  • Custom module реально центральный: recommendation engine объясним и project-specific.
  • Realtime, search, API, dashboard и social layer не декоративны — они встроены в продуктовый сценарий.
  • Команда понимает архитектуру, а не просто "подняла что-то, что работает на удаче".
